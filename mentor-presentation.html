Slide 02 / 07  ·  Concept

What is a REST API?

REST stands for Representational State Transfer. It is a way for two programs to talk to each other over HTTP — the same protocol your browser uses to load websites.

📞 Like a Phone Call

The browser (client) dials a number (URL).
The server picks up and answers with data (JSON).
The call ends — then starts again next time.

🍽️ Like Ordering Food

You (browser) tell the waiter (HTTP) what you want.
The kitchen (server) prepares it and sends it back.
You don't need to know how it was cooked.

Slide 04 / 07  ·  Code Walkthrough

Client vs Server — Code Side by Side

Tracing one complete request cycle: GET /api/dashboard

🌐 app.js — Browser (Client)
app.js — fetchDashboardData()
// 1. Browser calls fetch() with endpoint URL
async function fetchDashboardData() {
  try {

    const response = await fetch('/api/dashboard');

    // 2. Check the server sent back 200 OK
    if (!response.ok)
      throw new Error("Error: " + response.status);

    // 3. Parse JSON body into JavaScript object
    dashboardData = await response.json();

    // 4. Use the data to update the UI
    updateKPICards();
    renderAssetsTable();

  } catch (e) {
    // 5. If server offline → use local mock data
    dashboardData = localFallbackData;
  }
}
🐍 server.py — Python (Server)
server.py — do_GET() handler
# Server listens for incoming HTTP requests
class ClawSentinelAPIHandler(SimpleHTTPRequestHandler):

  def do_GET(self):

    # 1. Check which URL was requested
    if self.path == '/api/dashboard':

      # 2. Send HTTP 200 status header
      self.send_response(200)
      self.send_header(
        'Content-Type', 'application/json')
      self.end_headers()

      # 3. Convert Python dict → JSON string
      self.wfile.write(
        json.dumps(dashboard_data).encode())
1

Browser sends HTTP GET request

fetch('/api/dashboard') → opens a TCP connection to localhost:8000 and sends a GET request. No body needed for GET.

2

Server routes the request

Python's do_GET() fires. self.path equals '/api/dashboard' so the dashboard handler block runs.

3

Server serializes data to JSON

json.dumps(dashboard_data) converts the in-memory Python dictionary to a JSON string and writes it to the HTTP response body.

4

Browser parses JSON → updates UI

response.json() parses the string back into a JS object. Then updateKPICards(), renderAssetsTable() etc. refresh the dashboard.

Slide 05 / 07  ·  POST Deep-Dive

POST Requests — Sending Data to Server

Unlike GET, a POST request carries a JSON body — data sent from the browser to the server. Example: Deploy Mitigation Playbook.

🌐 app.js — executeMitigationPlaybook()
app.js
async function executeMitigationPlaybook(
  assetName, cve
) {
  const response = await fetch('/api/mitigate', {

    method: 'POST',         // ← Not GET!
    headers: {
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({   // ← Send data
      assetName,
      cve
    })
  });

  const data = await response.json();
  alert(data.message);
  renderAssetsTable();   // refresh UI
}
🐍 server.py — do_POST() /api/mitigate
server.py
def do_POST(self):
  # 1. Read how many bytes are in the body
  length = int(self.headers['Content-Length'])

  # 2. Read & parse the JSON body
  raw  = self.rfile.read(length).decode()
  body = json.loads(raw)

  if self.path == '/api/mitigate':
    name = body['assetName']
    cve  = body['cve']

    # 3. Mutate in-memory state
    for asset in dashboard_data['assets']:
      if asset['name'] == name:
        asset['status'] = 'in-progress'

    # 4. Return confirmation JSON
    self.send_response(200)
    self.end_headers()
    self.wfile.write(json.dumps({
      "success": True,
      "message": f"Playbook for {name} started"
    }).encode())
🔒
Offline Fallback — Production-Ready Pattern

Every fetch() call in app.js is wrapped in a try { } catch { } block. If the server is offline, the UI keeps working using the localFallbackData mock dataset in the browser. This shows fault-tolerant design — even with no server, the app still works.
